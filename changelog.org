#+TITLE: remoto.el changelog
#+AUTHOR: Ag Ibragimov

Changes to the project, following Keep a Changelog conventions.

* [1.8.1] - 2026-06-30

** Fixed
- Minibuffer completion of ~/github:~ and ~/gh:~ paths no longer doubles the prefix. The ~read-file-name-internal~ advice re-based candidates to full paths (for example the repo segment ~agzam/~ became ~/github:agzam/~), so base-aware completion frameworks - Vertico, and Emacs' default completion - prepended the directory a second time: completing ~/gh:agza~ inserted ~/gh:/gh:agzam/~ instead of ~/gh:agzam/~ (and ~/gh:agza/github:agzam/~ before that). The advice now injects only the completion metadata (category and affixation for the ~@~, ~#~, and owner/repo levels) and delegates completion to the original table, so candidates stay relative to the completion boundary and are inserted exactly once. The defect affected both the ~/gh:~ shorthand and the canonical ~/github:~ form and had been latent since the advice was introduced.

** Added
- A regression test that reproduces the doubling by mirroring how a base-aware framework inserts a candidate (~string[0..base]~ concatenated with the candidate), asserting a single-prefixed result for ~/gh:~ and ~/github:~ at the owner and nested-path levels.

* [1.8.0]

** Added
- Optional, forge-agnostic Embark integration (~remoto-embark.el~): ~embark-act~ on remoto targets (owner, repo, branch, issue, dir, file, and a forge ~url~ at point) in remoto buffers, ~dired~, and ~/github:~ minibuffer completion (including ~embark-collect~). It activates automatically when Embark is installed - an autoloaded ~(with-eval-after-load 'embark (remoto-embark-register))~ registers the keymaps the moment Embark loads, with no manual ~(require 'remoto-embark)~ (which still works for eager setup). Routing through the public ~remoto-embark-register~ instead of requiring the file avoids a load recursion when the file is loaded with Embark already present.

** Fixed
- Embark actions no longer fail with "no URL templates for forge nil" on an owner or other non-repository target reached through the repository or generic ~remoto~ keymap. The generic ~remoto-embark-copy-url~/~-browse-url~ now resolve a web URL for any target kind (issue/PR, repository, directory, file, or owner); the repository- and file-specific actions raise a clear "not a repository, directory, or file target" message instead of a cryptic forge-lookup failure.
- Owner targets now route to ~remoto-embark-owner-map~ rather than the repository keymap: a shared classifier (~remoto--embark-classify~) derives the Embark target type from the path shape (issue, branch, owner, or repo/dir/file).
- Embark actions now treat every input form for the same target identically. The ~/gh:~ shorthand, the file-name form ~/github:OWNER/REPO~ or ~/github:OWNER/REPO/~ (as typed at ~C-x C-f~), and the canonical ~/github:OWNER/REPO:/~ all resolve to the same target via ~remoto--embark-canonicalize~. Previously only the canonical colon form was understood, so ~embark-act~ on a typed ~/gh:agzam/mxp~ or ~/github:agzam/mxp~ failed.
- ~remoto--forge-url~ defaults a nil ref to ~HEAD~ (which forges resolve to the default branch), so the repository-level history action (~remoto-embark-copy-history-url~ on a repo target, which has no ref) builds a valid URL instead of erroring with "Invalid format character".

* [1.7.0] - 2026-05-31

** Added
- ~/gh:~ shorthand prefix as an alias for ~/github:~. ~C-x C-f /gh:OWNER/REPO@REF:/path~ now works exactly like the longer ~/github:~ form. The shorthand is an entry point only: it is normalized to the canonical ~/github:~ on the first completion or file operation, so paths still display and resolve canonically. The original ~/github:~ prefix is unchanged and fully backward compatible.
- ~remoto--normalize-shorthand~ - rewrites a leading ~/gh:~ in handler arguments to ~/github:~ so every downstream handler only ever sees canonical paths (no per-operation changes needed).
- ~remoto--shorthand-prefix~ - the ~/gh:~ alias prefix constant.
- Unit tests covering the normalizer (shorthand rewrite, canonical pass-through, non-string and unrelated-string pass-through), the widened ~remoto--handler-regexp~ (matches ~/gh:~ and ~/github:~, rejects look-alikes such as ~/ghostly:~), and ~/gh:~ dispatch parity with ~/github:~ for ~file-directory-p~, ~file-exists-p~, ~file-remote-p~ (prefix and method), ~file-name-directory~, and the public file API.

** Changed
- ~remoto--handler-regexp~ widened to match both ~/github:~ and the ~/gh:~ shorthand, so the ~/gh:~ entry is routed through ~file-name-handler-alist~ to the same handler.

* [1.6.0] - 2026-05-30

** Added
- In-flight fetch indicator: while completion data loads (for example the org repo list at ~/github:OWNER/~, or while ~remoto-browse~ searches GitHub), a ~[fetching...]~ marker appears in the minibuffer. It is drawn as minibuffer text, so it works with any completion UI (vertico, icomplete, default). Toggle with ~remoto-show-fetch-indicator~ (default t).
- ~remoto-browse~ shows the indicator in the echo area during the synchronous resolve/open that runs after a repo is selected, so the blocking GitHub round-trips are no longer silent.
- ~remoto-show-fetch-indicator~ - customization to enable or disable the indicator (now governs both the completion overlay and the ~remoto-browse~ echo-area indicator).
- ~remoto--invalidate-completion-ui~ - refreshes an in-progress minibuffer completion across front-ends after async data arrives.
- Unit tests covering the in-flight counter (single decrement on success, error, and synchronous failure; correct behavior with overlapping fetches), the indicator overlay lifecycle and its cursor-preserving text property, the indicator appearing for ~remoto-browse~ (both completion and the synchronous fetch), the completion-refresh invalidation, and the consolidated repo search delegating to the shared owner-repos engine.
- ~remoto--search-owner-repos-sync~ and ~remoto--recent-owner-repos-sync~ - synchronous owner-repo fetches wrapped in ~while-no-input~. The ~/github:OWNER/~ completion branch uses them as a cold-cache fallback so the first computation can return candidates instead of depending solely on the async refresh landing.
- ~remoto--extract-search-repos~ - shared helper that builds propertized repo candidates (name plus description) from a GitHub search response, replacing three duplicated inline mappers.
- Unit tests for the synchronous fallbacks (success and error paths), the owner-level sync-fallback wiring (and that it is skipped below ~remoto-min-search-chars~ or when async already has results), the trailing-slash repo candidates, the shared extractor, and the ~remoto~ category registration.

** Fixed
- Minibuffer cursor no longer jumps when the indicator renders: the ~[fetching...]~ after-string carries a ~cursor~ text property, so point stays at the end of the typed input instead of being drawn past the marker.
- ~remoto-browse~ now finds all of an organization's matching repositories instead of only those in the first page of the owner listing. Its repo search previously derived results from a capped ~user:OWNER~ listing (via ~remoto--search-repos-from-parent~) and never issued the precise ~QUERY in:name user:OWNER~ query, so repos beyond the first page (for example ~qlik-trial/stitch-*~) never appeared. It now delegates to the same ~remoto--search-owner-repos~ engine the ~/github:~ completion uses.
- Completion candidates now appear automatically when an async fetch completes. Previously the list stayed empty until the input was edited (for example deleting and retyping the trailing slash), because vertico skips recomputation while the typed input is unchanged. The refresh now voids vertico's input cache and the default sorted-completions cache, runs ~post-command-hook~ for icomplete/fido, and rebuilds ~*Completions*~ when it is displayed.
- ~remoto--api-async~ decrements its in-flight counter exactly once even though ~ghub~ retries the callback with fewer arguments on an arity mismatch. Without this the indicator could clear while other fetches were still running.
- Clean byte-compilation: ~remoto-search-cache-ttl~ was referenced (in ~remoto--fetch-dir-children-light~) before its ~defcustom~, producing a free-variable warning that failed ~make check-compile~. The defcustom is now defined ahead of its first use.
- Repo candidates at ~/github:OWNER/~ now carry a trailing ~/~, matching the owner candidates at ~/github:~. Selecting a repo descends into it (default-branch file listing) instead of re-running a repo search on the bare name.
- Owner-level repo completion no longer returns an empty candidate list while the async search is still in flight on a cold cache: the branch falls back to a synchronous ~while-no-input~ fetch (cached on completion) and only yields when the user keeps typing. Note: this addresses the cold-cache empty return at the function level; the broader async-refresh-vs-sync timing in the live minibuffer is not yet fully characterized.
- The ~remoto~ completion category is registered through ~completion-category-overrides~ instead of ~completion-category-defaults~. With a global ~orderless~ in ~completion-styles~, the defaults entry was bypassed (and could be reset elsewhere), so owner-level matching fell back to orderless substring behavior instead of the intended ~partial-completion~/~basic~ prefix filtering.
- Clean byte-compilation, second instance: ~remoto-min-search-chars~ is now forward-declared ahead of ~remoto--handle-file-name-all-completions~, which references it before the ~defcustom~.

* [1.5.3] - 2026-05-30

** Fixed
- Branch, tag, and organization completion now return the full result set instead of only the first 100 entries. Large repositories (e.g. NixOS/nixpkgs with >100 branches) previously showed a truncated list because each fetch issued a single ~per_page=100~ request.

** Added
- ~remoto--paginated-api~ - follows pagination to fetch all pages of a GitHub endpoint, bounded by an optional ~max-pages~ cap (default 10) so very large repositories cannot trigger an unbounded run of synchronous API calls.
- Unit tests for ~remoto--paginated-api~ covering multi-page accumulation, the page cap, and the query-string separator.

* [1.5.2] - 2026-05-08

** Fixed
- Always pass ~:host "api.github.com"~ to ~ghub-get~ in all call sites (~remoto--ghub-get~, ~remoto--api-async~). The conditional guard only sent it for string auth tokens, so unauthenticated calls (~auth = 'none~) hit ~github.com~ instead of the API, returning HTML that crashed the JSON parser.
- ~remoto--json-reader~ returns nil instead of signaling on malformed JSON, preventing unhandled errors in async contexts where no ~condition-case~ protects the reader callback.
- ~remoto--warm-auth~ no longer permanently disables auth when the idle-timer token lookup fails. Previously, a transient failure (GPG agent not ready, ghub not loaded) set ~remoto--auth-failed~ to t, locking out private repos for the entire session.
- ~remoto--api~ and ~remoto--api-async~ now try ~remoto--find-github-token~ directly when no cached token or user config exists, instead of relying solely on ghub's built-in auth-source lookup (which uses different host/user patterns and often fails on fresh sessions). Found tokens are cached in ~remoto--effective-auth~.
- Auth failures now signal ~user-error~ with a clear message instead of silently falling back to unauthenticated access. Previously, private repos would quietly show empty results with only a message in ~*Messages*~ that was easily missed.

** Added
- Full pipeline tests asserting every segment (API, default-branch, fetch-tree, tree-entry, tree-children, file-exists-p, file-directory-p, file-regular-p, directory-files, file-attributes, insert-directory) returns non-nil data.
- Auth forwarding tests verifying the token reaches ghub-get and that auth=none is used when auth-failed is set.
- Fresh-session auth resolution tests: fallback to ~remoto--find-github-token~ when ghub fails, loud ~user-error~ when no token exists anywhere.
- JSON reader unit tests covering headers, HTML error pages, empty buffers.

* [1.5.1] - 2026-05-07

** Fixed
- Auth on fresh Emacs sessions. ghub's token resolution searched ~github.com~ while tokens were stored under ~api.github.com~, causing all API calls to fail with a cryptic JSON parse error. Remoto now searches auth-source directly, trying both hosts with ~^forge~, ~^ghub~, and bare username suffixes, then passes the token as a literal string with an explicit ~:host~.
- ~remoto--warm-auth~ bypasses ~remoto--api~ timeout entirely - calls ghub directly so GPG passphrase entry can complete naturally (like forge does).
- ~remoto--ghub-get~ catches ~json-error~ and translates it to a readable ~user-error~ instead of letting it propagate as a raw process filter crash.

** Changed
- ~remoto-auth-timeout~ default raised from 2 to 10 seconds - 2s was too aggressive for first-time GPG decryption.

** Added
- ~remoto--find-github-token~ - searches auth-source across host and package-suffix combinations, finding the first available token regardless of how it was stored.
- ~remoto--effective-auth~ - caches the resolved token so subsequent API calls skip the probe. Cleared by ~remoto-reset-auth~.

* [1.5.0] - 2026-05-06

** Changed
- Completion for user/org search, repo search, and browse-repos is now fully non-blocking. HTTP requests are debounced and fired asynchronously via ~ghub~ callbacks; cached results are returned immediately while fresh data loads in the background.
- Minimum search character threshold raised from 2 to 3 (configurable via ~remoto-min-search-chars~).
- Repo list cache uses a separate longer TTL (~remoto-repo-cache-ttl~, default 30 min) since repo lists rarely change.
- ~remoto--prefetch-owner-repos~ now uses truly async HTTP instead of ~while-no-input~ wrapper.
- Repo search below the minimum character threshold narrows from the recent-repos cache instead of returning nil.
- ~remoto--search-repos-from-parent~ uses substring matching instead of prefix-only when filtering repo names.
- ~remoto--json-reader~ scans for ~{~ or ~[~ instead of relying on blank-line header heuristic, fixing compatibility across ghub/Emacs versions.

** Added
- ~remoto-debounce-delay~ - configurable idle time (default 0.3s) before firing search requests.
- ~remoto-min-search-chars~ - configurable minimum characters before searching users/repos (default 3).
- ~remoto-repo-cache-ttl~ - longer cache TTL for repo lists (default 1800s).
- ~remoto--api-async~ - async GitHub API calls via ~ghub-get~ with ~:callback~.
- ~remoto--debounce~ - key-aware idle timer that prevents completion frameworks from starving the async fetch.
- ~remoto--refresh-minibuffer-completions~ - framework-agnostic minibuffer refresh after async cache updates.
- ~remoto--search-cache-get~ now accepts an optional TTL parameter for per-key TTL overrides.
- Fail-fast for integration tests (stop after first failure, skip remaining).
- CI: split unit and integration test jobs, nix/elpa caching, ~workflow_dispatch~ trigger, PAT-based authenticated integration tests.

** Fixed
- ~files-default~ handler catches 404 from non-existent repos instead of crashing vertico.
- ~remoto--api-async~ auth: pass nil (not ~'none~) to ghub for default token resolution.
- Suppress ~auth-source-search~ echo messages during async completion.

* [1.4.0] - 2026-05-06

** Added
- Three new completion levels: files-default (~/github:o/r/~), issues (~/github:o/r#~), branches+tags (~/github:o/r@~).
- Root level (~/github:~) shows authenticated user + org memberships when query is empty.
- ~remoto-topic.el~ - dedicated issue/PR display buffers. PR-aware rendering with diff stats, branch info, merge state, and review summary.
- Completion annotations at every level via ~affixation-function~: repo descriptions, issue/PR titles, branch/tag grouping, user/org type, per-file last commit messages.
- ~remoto--fetch-tags~ for tag completion at the ~@~ level.
- ~remoto--fetch-issues~, ~remoto--search-issues~, ~remoto--fetch-issue~ for ~#~ level.
- ~remoto--fetch-issue-comments~ for topic display.
- ~remoto--fetch-file-commits~ for per-file commit message annotations (capped at 20 per directory).
- ~remoto--fetch-user-orgs~ using authenticated ~/user/orgs~ endpoint.
- ~remoto-annotation~ face (inherits ~completions-annotations~).
- Annotation alignment via ~(space :align-to N)~ display property.
- ~remoto-topic-mode~ (derived from ~special-mode~) with ~b~ (browse) and ~g~ (refresh) keybindings.

** Changed
- Completion category set to ~remoto~ (with ~partial-completion~ style) when annotations are active, preventing Marginalia from overriding.
- PRs sorted before issues in ~#~ completion.
- ~remoto--parse-partial-canonical~ excludes ~#~ from owner/repo character class.
- ~remoto-unload-function~ now clears all 9 caches.

** Fixed
- ~find-file-noselect~ on ~/github:o/r#42~ now correctly routes to topic display instead of treating ~#42~ as part of the repo name.
- ~remoto--maybe-rewrite~ no longer mangles ~#NUM~ paths.
- Carriage returns (~^M~) stripped from issue/PR body and comments.

* [1.3.0] - 2026-04-28

** Changed
- Repo completion now uses the GitHub search API instead of pre-fetching all repos. At ~/github:owner/~ with empty input, the 30 most recently updated repos are shown. As the user types, matching repos are fetched via ~search/repositories~. This replaces the old approach that fetched every repo for an owner (thousands for large orgs), which caused hangs.
- Auth timeout in ~remoto--api~ now only applies on the first call before the authenticated user is known. Once auth succeeds, the timeout is skipped so slow API responses (e.g. search) are not aborted.
- HTTP errors (404, 403) from the GitHub API no longer poison ~remoto--auth-failed~. Only actual auth-source/token errors trigger the unauthenticated fallback.

** Added
- ~remoto--search-owner-repos~ for searching repos within an owner via the search API, with caching and client-side narrowing for longer queries.
- ~remoto--recent-owner-repos~ for fetching the 30 most recently pushed repos of an owner.
- Non-blocking completion: all pre-repo completion levels (user search, repo listing, branch listing) are wrapped in ~while-no-input~ so typing interrupts pending API calls instead of freezing Emacs.
- Speculative pre-fetch: when user search returns results, an idle timer pre-fetches repos for the top match so the cache is warm by the time the user types ~/~.
- ~file-accessible-directory-p~ handler for remoto paths.
- ~read-file-name-internal~ advice so ~/github:~ completion works correctly with Vertico/orderless when ~default-directory~ prefixes are present.
- ~file-exists-p~ now returns t for ~/github:owner/repo~ partial paths (no ~@~), fixing completion transitions.

** Fixed
- ~expand-file-name~ no longer crashes on partial ~/github:~ paths (e.g. ~/github:owner/~ or ~/github:owner/repo@~). This broke ~embark-collect~ and ~embark-export~ at the repo and branch completion levels.
- ~remoto--api~ now falls back to unauthenticated access when ghub raises a ~user-error~ about missing username config (e.g. ~github.user~ unset). Previously this was re-raised as an API error, crashing instead of falling through to the unauthenticated path.

** Removed
- ~remoto--fetch-user-repos~ and ~remoto--user-repos-cache~. The "fetch all repos then filter client-side" approach is replaced by search API calls.

* [1.2.0] - 2026-04-27

** Added
- Private repository listing for the authenticated user. When browsing your own repos at ~/github:yourname/~, remoto now uses the ~/user/repos?type=owner~ endpoint which includes private repositories. Other users' repos continue using the public ~/users/{owner}/repos~ endpoint.
- ~remoto--get-authenticated-user~ to detect and cache the logged-in GitHub user via the ~/user~ API.

** Fixed
- Branch completion now works with orderless/vertico. The ~@~ in ~/github:owner/repo@~ is treated as a directory boundary so completion frameworks that fetch all candidates before filtering (e.g. orderless) get branch names instead of the repo list.

* [1.1.0] - 2026-04-26

** Added
- ~find-file~ completion for ~/github:~ paths. Type ~/github:foo~ to search users/orgs, ~/github:owner/~ to list repos, append ~@~ after a repo name to complete branches. Transitions seamlessly into file-level completion within a repo.
- ~remoto--search-users~ for searching GitHub users/orgs via the search API with caching and client-side narrowing.
- ~remoto--fetch-user-repos~ for fetching repos of a user/org via the GitHub API with caching.
- Partial path support in file operations (~file-exists-p~, ~file-directory-p~, ~file-name-directory~, ~file-name-nondirectory~, ~file-remote-p~) for pre-repo completion levels.
- ~remoto--parse-partial-canonical~ for resolving ~/github:owner/repo~ and ~/github:owner/repo@ref~ paths when opening.

* [1.0.0] - 2026-04-25

** Changed
- Replaced ~gh~ CLI dependency with the ~ghub~ Emacs library for all GitHub API access. No external binary needed - authentication is handled through ~auth-source~.
- Added ~remoto-github-auth~ defcustom for controlling which ghub token to use.
- Public repos now work without any token configuration (falls back to unauthenticated access at 60 req/hr).
- Search cache entries now expire after 5 minutes (~remoto-search-cache-ttl~), preventing stale results from persisting across an entire session.
- Bare search queries (no ~owner/~ prefix) now search by repository name instead of GitHub username, so typing "remoto.el" finds repos named "remoto.el".

** Fixed
- ~remoto--search-repos~ now accepts an optional callback argument for compatibility with consult's async dynamic collection protocol.
- Git remote URLs with dotted repo names (e.g., ~git@github.com:agzam/remoto.el.git~) now parse correctly.
- Search cache now correctly caches empty results instead of re-hitting the API on every query.

** Removed
- ~gh~ CLI is no longer required. Removes implicit telemetry that ~gh~ collects unless users explicitly opt out.

* [0.1.0] - 2026-04-24

** Added
- Initial implementation of virtual filesystem for GitHub repos.
- Path parser for canonical ~/github:owner/repo@ref:/path~ format.
- User-facing URL parser accepting GitHub URLs, git remotes, owner/repo shorthand.
- Tree cache using Git Trees API (single call per repo).
- File content fetching via Contents/Blobs API with sha-based caching.
- Dired integration with ~dired-subtree~ compatibility.
- Read-only enforcement for all write operations.
- ~remoto-browse~ command as the main entry point.
- ~remoto-refresh~ command for cache invalidation.
- ~remoto-copy-github-url~ for reverse URL mapping.
- Transparent ~dired~ / ~find-file~ advice for GitHub URL rewriting.
